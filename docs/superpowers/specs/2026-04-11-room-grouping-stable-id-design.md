# Группировка створок по комнатам: переход на стабильный room-ID

**Статус:** черновик · **Дата:** 2026-04-11

## Проблема

В форме заказа (`client/src/pages/orders/order-form.tsx` + `use-room-groups.ts` + `room-container.tsx`) створки группируются в «комнаты». Сейчас логика привязана к тому, что **имя комнаты** одновременно является её идентификатором. Это вызывает два связанных дефекта:

1. В `useRoomGroups.buildRooms()` всегда принудительно добавляется пустая «дефолтная» комната с именем `""` — она не исчезает, даже если в ней нет створок.
2. При попытке переименовать эту дефолтную комнату (двойной клик → ввести «Кухня»):
   - У всех створок с `roomName === ""` обновляется `roomName` на `"Кухня"`.
   - В `createdRooms` появляется новая строка `"Кухня"`.
   - Но `buildRooms()` снова добавляет пустой дефолт первым — и визуально выглядит так, будто появилась новая «Кухня», а первая комната осталась пустой.

Результат: UI показывает две комнаты (одна пустая, одна именованная) вместо одной переименованной. Пользователь воспринимает это как баг добавления комнаты.

## Цели

- Убрать появление «клона» при переименовании первой комнаты.
- Сохранить привычный UX переименования через двойной клик по названию.
- Не трогать схему БД (использовать то, что уже есть).
- Не ломать отображение и сохранение старых заказов.
- Не ломать импорт заказа из мобильного замера.

### Не-цели

- Менять схему `order_sashes` в БД.
- Менять кнопку «+ Комната» на inline-инпут или другой способ добавления.
- Вводить обязательные валидации на имя комнаты (пустые имена остаются допустимыми).
- Переделывать визуальный стиль `RoomContainer`.

## Корневая причина

Группировка в `buildRooms()` идёт по `sash.roomName`. При этом в схеме `order_sashes` (и в zod-схеме `sashSchema`) уже есть целочисленное поле `room`, но оно не используется для группировки. В результате имя — это и display, и primary key одновременно, а значит любой rename ломает идентичность группы.

## Решение

Перевести группировку на стабильный числовой `room` ID. Имя (`roomName`) становится только display-значением, может быть пустым, может повторяться между комнатами, свободно редактируется без влияния на идентичность группы.

### Модель данных

В схеме БД ничего не меняется. В `shared/schema.ts` у `orderSashes` уже есть:

- `room: integer("room").default(1)` — станет стабильным ID комнаты внутри заказа.
- `roomName: text("room_name")` — остаётся display-именем, может быть пустым/null.

В zod-схеме `sashSchema` поле `room` уже объявлено как `z.number().optional().default(1)`, `roomName` — `z.string().optional()`. Никаких миграций не требуется.

### Состояние хука `useRoomGroups`

Вместо `createdRooms: string[]` хранить:

```ts
interface Room {
  id: number;    // стабильный ID, уникальный в пределах формы
  name: string;  // display-имя, "" = безымянная
}
const [rooms, setRooms] = useState<Room[]>([]);
```

`buildRooms()` собирает группы **по `sash.room`**, а не по `sash.roomName`. Для каждой группы имя берётся из первой створки группы (`roomName || ""`) — внутри одной группы `roomName` у всех створок всегда одинаковый, потому что переименование идёт пакетно.

Принудительное добавление дефолтной пустой комнаты в `buildRooms()` удаляется.

### API хука

Публичные методы меняют сигнатуру — теперь оперируют room-ID, а не именами:

- `addRoom(): number` — создаёт новую комнату с уникальным ID (`max(ids) + 1`) и пустым именем, добавляет одну пустую створку с этим `room`, возвращает новый ID. Имя не принимается — хедер новой комнаты автоматически стартует в режиме редактирования (см. раздел «UX-взаимодействия»).
- `renameRoom(roomId: number, newName: string): void` — обновляет `rooms[i].name` для комнаты с этим ID, пакетно обновляет `sash.roomName` у всех створок где `sash.room === roomId`.
- `removeRoom(roomId: number, moveSashesTo?: number | null): void` — удаляет комнату из `rooms`. Если `moveSashesTo` задан — меняет `sash.room` для затронутых створок. Если `moveSashesTo === null` — удаляет створки из массива. Если у комнаты нет створок — параметр не требуется.
- `moveSash(sashIndex: number, targetRoomId: number): void` — обновляет `sash.room` у указанной створки.

### Инициализация `rooms` от текущих створок

Хук должен пересоздавать `rooms` в трёх случаях:

1. **Новый заказ** — после `form.reset()` на создание. В массиве одна пустая створка с `room: 1`. Хук строит одну комнату `{id: 1, name: ""}`.
2. **Редактирование существующего заказа** — перед `form.reset()` вызывающий код (`index.tsx`) прогоняет створки через нормализацию (см. раздел «Переход со старых данных»), затем хук строит `rooms` по полученным `sash.room` и `sash.roomName`.
3. **Импорт из замера** (`openFromMeasurement`) — вызывающий код применяет ту же нормализацию перед `form.reset()`.

Чтобы хук корректно пересобирал `rooms` после `form.reset()`, использовать явный метод `seedRooms()`, который вызывается `OrderForm` в ответ на увеличение `resetToken` — числового пропа, который `index.tsx` инкрементирует перед каждым `form.reset()`. `OrderForm` слушает `useEffect([resetToken])` и вызывает `seedRooms()`. Такая схема не зависит от того, совпала ли длина `fields` до и после reset.

Внутри хука для отслеживания изменений `sashes.room` (при `moveSash`, `addRoom`, `removeRoom`) использовать `useWatch({ control, name: "sashes" })` — это гарантирует, что любое изменение `sash.room` в форме вызывает перерасчёт групп в `useRoomGroups` без ручного `bump()`.

### Переход со старых данных

На стороне БД миграция не нужна. На стороне чтения заказа (`openEditDialog`) сделать нормализацию — единое правило, работающее во всех случаях:

1. Для каждой створки определить «ключ группы»: `s.roomName || ""`.
2. Построить map `ключ → номер` (первое уникальное значение → 1, второе → 2, и т. д.).
3. Перезаписать `sash.room` по этому map'у перед вызовом `form.reset()`.

Результат:
- Все створки без имени (`roomName === "" | null`) попадают в одну группу `room=1`, показываются как одна безымянная комната.
- Створки с одним и тем же именем попадают в одну группу, независимо от того, какие `room` integer у них были до этого.
- Створки с разными именами получают разные `room` integer.

Эта нормализация нужна только при входе в форму редактирования и при импорте из замера (`openFromMeasurement`). Сохранение наружу идёт уже с нормализованным `sash.room`, так что после одного сохранения данные в БД становятся консистентными.

### UX-взаимодействия

Все визуальные элементы `RoomContainer` остаются как сейчас, кроме того, что `roomName` и `droppableId` заменяются на `roomId`:

- **Отображение безымянной комнаты** — плейсхолдер `«Название комнаты (двойной клик)»` серым курсивом (как сейчас).
- **Двойной клик по названию** — переход в режим редактирования с `autofocus` (как сейчас).
- **Enter/`blur`** — `onRenameRoom(roomId, trimmed)`. Если имя не изменилось, режим редактирования закрывается без действия.
- **`+ Комната`** — теперь сразу вызывает `addRoom()` (без popover с инпутом). Новая комната добавляется в список, получает новый ID, в неё добавляется одна пустая створка, и её хедер автоматически переходит в режим редактирования с `autofocus` (чтобы пользователь сразу начал печатать имя). Реализация: `OrderForm` хранит `autoEditRoomId: number | null`, который после `addRoom()` равен новому ID; `RoomContainer` с совпадающим `room.id` видит проп `autoEdit={true}` и в `useEffect` один раз переключается в режим редактирования. После первого коммита или `blur` — значение сбрасывается. Это делает поведение консистентным с двойным кликом, но избавляет от необходимости двойного клика для только что созданной комнаты.
- **🗑 пустой комнаты** — удаляется без подтверждения.
- **🗑 последней комнаты** — кнопка неактивна (всегда нужна хотя бы одна комната).
- **🗑 комнаты со створками** — открывается диалог с:
  - Заголовок: «Удалить комнату «{name || "без названия"}»?»
  - Текст: «В ней {n} створок. Выбери что с ними делать:»
  - Radio-список: «Перенести в {другая_комната}» для каждой другой комнаты + «Удалить вместе со створками» (красным).
  - Кнопки: «Отмена» и «Применить».
  - По применению вызывается `removeRoom(roomId, targetRoomId)` или `removeRoom(roomId, null)`.
- **Drag-and-drop** — `droppableId` берётся из `room-${id}`. `moveSash(index, targetRoomId)` при drop.
- **Сворачивание комнаты** — локальный state в `RoomContainer`, без изменений. Ключ инстанса — `room-${id}`, чтобы состояние сохранялось при переупорядочивании/переименовании.

### Компонентные изменения

**`use-room-groups.ts`** — переписывается:

- Внутреннее состояние `rooms: Room[]`.
- `useWatch` на `sashes` — чтобы любое изменение `sash.room` (через `moveSash`, `addRoom`, `removeRoom`) вызывало перерасчёт групп без ручного триггера.
- `buildRoomGroups()` — возвращает `{ room: Room; sashIndices: number[] }[]`, вычисляется из `rooms` + подписанных `sashes` каждый рендер.
- Метод `seedRooms(sashes)` для явной синхронизации после `form.reset()`: строит `rooms` из переданных сашек и сбрасывает внутренний state.
- Методы `addRoom`, `renameRoom`, `removeRoom`, `moveSash` — с ID вместо имён.
- Старый `bump()` и `version`-state удаляются.

**`order-form.tsx`**:

- Убирается локальное состояние `showAddRoom`, `newRoomName` и соответствующий `Popover`. Кнопка «+ Комната» теперь напрямую вызывает хуковый `addRoom()` и выставляет `autoEditRoomId` равным возвращённому ID.
- Принимает новый проп `resetToken: number` от `index.tsx`, слушает его в `useEffect` и вызывает `seedRooms(form.getValues("sashes"))`.
- `RoomContainer` получает `roomId`, `roomName`, `autoEdit`, `canDelete` вместо старых пропов.
- `onRenameRoom(newName)` → вызывает `renameRoom(roomId, newName)` из хука.
- `onDeleteRoom()` → проверяет наличие створок: если пусто, сразу `removeRoom(roomId)`; если нет — открывает локальную state-модалку `{ roomId, sashCount } | null`.
- `handleDragEnd` извлекает `targetRoomId` из `over.id` (`room-${id}`).
- При добавлении створки (`onAddSash` в `RoomContainer`) `room` проставляется из ID контейнера.

**`room-container.tsx`**:

- Пропы `roomName: string, droppableId: string, isDefault: boolean` → `roomId: number, roomName: string, autoEdit: boolean, canDelete: boolean`. Остальные пропы (`sashIndices`, `fields`, `form`, `systems`, `fabrics`, `totalFields`, `onRemoveSash`, `onRenameRoom`, `onDeleteRoom`, `onAddSash`, `calculatingSashes`) — без изменений.
- `droppableId` формируется как `room-${roomId}` внутри `RoomContainer`.
- Отображение имени: если `roomName === ""` — плейсхолдер, иначе — жирный текст. Убрать `isDefault` как отдельный концепт (пустое имя не означает «дефолтная» — это просто безымянная обычная комната).
- Кнопка 🗑 показывается всегда, но `disabled`, если `canDelete === false` (комната единственная в форме).
- `autoEdit={true}` на первом рендере включает режим редактирования один раз (через `useEffect([autoEdit])`), после чего родитель должен очистить свой `autoEditRoomId`.

**Новый компонент `delete-room-dialog.tsx`** (или inline в `order-form.tsx`):

- Модалка с radio-списком для выбора «куда перенести створки». Скромный UI: один заголовок, список, две кнопки.

**`index.tsx` (страница `orders`)**:

- Хранит state `resetToken: number` и инкрементирует его перед каждым `form.reset()` (в `resetForms`, `openEditDialog`, `openFromMeasurement`). Передаёт в `<OrderForm resetToken={resetToken} />`.
- `resetForms()` — первая створка получает `room: 1`.
- `openEditDialog()` — перед `form.reset()` применяет нормализацию `sash.room` по группам `roomName` (см. раздел «Переход со старых данных»). Затем инкрементирует `resetToken`.
- `openFromMeasurement()` — то же самое: нормализация + инкремент `resetToken`.
- В `onSubmit` ничего не меняется: `room` и `roomName` уходят в БД как есть; размножение по `quantity` копирует оба поля.

## Что НЕ меняется

- Поля створки: размеры, система, ткань, коэффициент, цена — без изменений.
- Drag-and-drop механизм (`@dnd-kit`) — без изменений, только меняется формат `droppableId`.
- Расчёт себестоимости и цены — без изменений.
- Подробности заказа (`cost-calculation-dialog`) — без изменений.
- Сохранение заказа в БД и раздача по створкам (`quantity > 1`) — без изменений.

## Критерии успеха

1. На новой форме заказа — одна комната с плейсхолдером, одна пустая створка. Дефолтной-вечной комнаты нет.
2. Двойной клик → «Кухня» → Enter: комната переименована в месте, пустой клон не появляется.
3. «+ Комната» → появляется вторая комната с автофокусом на инпуте имени.
4. 🗑 комнаты с створками → модалка выбора «куда перенести»; один из вариантов — удалить все вместе.
5. 🗑 единственной комнаты — кнопка неактивна.
6. Drag-and-drop между комнатами работает как раньше.
7. Открытие старого заказа (у которого все створки `room: 1, roomName: null`) → одна безымянная комната со всеми створками.
8. Открытие старого заказа (у которого створки с разными `roomName`) → столько комнат, сколько уникальных имён. После сохранения `sash.room` становится консистентным.
9. Заказ из мобильного замера: если в замере были комнаты с разными именами — в заказ они приходят в разных группах.
10. После сохранения и повторного открытия состояние групп не меняется.
